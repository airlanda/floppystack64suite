FS64 Angular Interop Summary

This document summarizes the two Angular interop patterns tested against the FloppyStack micro front ends: a React bridge for disks and a Web Component wrapper for games. It uses the current code in the monorepo and explains runtime relationships, data flow, lifecycle, and tradeoffs.

3. Method A: React Bridge for Disks

How it works
  • Angular loads disks from the manifest.
  • Angular requests the exposed module ./AngularBridge.
  • The remote exports imperative functions to mount and unmount the React tree.
  • The React app is wrapped in MemoryRouter so Angular does not need to hand off browser routing.
Why this pattern is useful
  • Works well for a large React feature slice.
  • Keeps Angular in control of page composition.
  • Gives explicit lifecycle control for mount and teardown.
  • Good when the embedded React UI is more like a mini app than a widget.

Key Angular usage

async ngAfterViewInit() {
  this.bridge = await this.loader.loadRemoteById<DisksBridgeModule>(
    'disks',
    './AngularBridge'
  );

  this.bridge.mountFs64DisksBridge(this.mountPoint.nativeElement, {
    initialPath: '/?diskId=1',
  });
}

Key React bridge export

export function mountFs64DisksBridge(element: Element, options?: { initialPath?: string }): void {
  const root = createRoot(element);
  root.render(
    <React.StrictMode>
      <MemoryRouter initialEntries={[options?.initialPath || '/']}>
        <App />
      </MemoryRouter>
    </React.StrictMode>
  );
}

4. Method B: Web Component for Games

How it works
  • Angular loads games from the manifest.
  • Angular requests the exposed module ./WebComponent.
  • The remote defines <fs64-games-widget>.
  • Angular renders the element directly and updates it through an attribute.
Why this pattern is useful
  • Best for a self-contained widget surface.
  • The host only needs DOM-level integration.
  • Simple mental model for other frameworks too, not just Angular.
  • Good when the remote needs a small input API and little host lifecycle control.

Key Angular usage

<fs64-games-widget class="interop-surface" [attr.query]="query"></fs64-games-widget>

async ngAfterViewInit() {
  const module = await this.loader.loadRemoteById<GamesElementModule>(
    'games',
    './WebComponent'
  );
  module.defineFs64GamesElement();
}

Key custom element export

class Fs64GamesElement extends HTMLElement {
  static get observedAttributes() {
    return ['query'];
  }

  private renderReactTree() {
    const query = this.getAttribute('query') || '';
    const initialPath = query ? `/?q=${encodeURIComponent(query)}` : '/';

    this.root.render(
      <MemoryRouter initialEntries={[initialPath]}>
        <App />
      </MemoryRouter>
    );
  }
}

5. Shared Dynamic Loader

Both methods use the same manifest-driven remote resolution. The Angular host does not hardcode remote URLs in the page component. It asks FederationLoaderService for a remote by id, and that service resolves the runtime manifest from /fs64-remotes.json. 

async loadRemoteById<T>(id: string, exposedModule = './Module'): Promise<T> {
  const remote = await this.getRemoteById(id, exposedModule);
  return loadFs64RemoteModule<T>(remote);
}

7. Tradeoffs

MethodStrengthsCostsBest fit
React bridge Explicit lifecycle, easy to mount a full React app, host controls teardown. Imperative API, more host code, not as HTML-native. Large feature slice like disks.
Web Component Framework-neutral, host template stays simple, easy input surface. DOM attribute boundary is more limited, complex interactions can get awkward. Widget-like slice such as games search.

8. Current Caveats

  • Angular dev deep links are still weaker than the root route. The reliable path in dev is to open http://localhost:4300/ and navigate from the Angular shell.
  • The shared loader now tolerates incomplete host share initialization for this interop experiment. That is acceptable for a test path, but a production Angular host should be a clean federation host with an explicit host runtime strategy.
  • Backend CORS had to be extended for :4300 so the embedded remotes could fetch live data.

Draft: Multi-Framework Micro Front End Architectures

This draft frames micro front ends as both an architectural strategy and an implementation problem, with special attention to heterogeneous frontend environments where different teams or products use different frameworks. The paper is grounded in a practical case study based on the FS64 workspace, where React remotes are consumed both by a React host and by an Angular host through two different interoperability patterns.

2. Draft Abstract

Micro front end architectures are often presented as a natural extension of backend microservices, promising independent deployment, team autonomy, and improved scalability of frontend development. In practice, however, many organizations operate in heterogeneous frontend environments where framework diversity is unavoidable rather than intentional. This raises a harder architectural question: how should micro front ends be designed, integrated, and evaluated when hosts and remotes do not share the same framework runtime?

This paper examines multi-framework micro front end architectures from both an architectural and implementation perspective. It first surveys the major composition strategies used in industry, including static and dynamic module federation, framework bridges, Web Components, and host-controlled runtime manifests. It then evaluates these strategies using a concrete case study based on the FS64 workspace, where React micro front ends are loaded by both a React host and an Angular host. Two cross-framework integration strategies are compared in detail: an imperative React bridge for a large remote and a Web Component wrapper for a widget-style remote.

The analysis focuses on performance, developer experience, reusability, routing ownership, operational complexity, and backend integration. The paper argues that the principal difficulty in multi-framework micro front ends is not simply remote loading, but the definition of a robust boundary between architectural independence and runtime interoperability. It concludes with practical guidance for choosing between bridge-based, Web Component-based, and manifest-driven federation strategies depending on the size, autonomy, and coupling of the remote.

6. Case Study Framing for FS64

FS64 is a strong case study because it contains both architectural and implementation diversity in a single workspace:

  • React-based MFEs with dynamic manifest-driven runtime loading.
  • An Angular host consuming those remotes through two different interop boundaries.
  • A shared Express backend serving API and metadata endpoints.
  • Cross-framework integration issues that are realistic rather than hypothetical: share-scope initialization, host runtime mismatch, CORS, and route ownership.
The paper should emphasize that the difficult part of multi-framework MFEs is not merely loading code from another bundle. It is defining an integration boundary that is operationally stable, development-friendly, and reusable across hosts.

7. Suggested Methodology

  1. Describe the architectural patterns under comparison.
  2. Implement each pattern in the same monorepo so the backend and domain remain constant.
  3. Measure runtime and developer-facing characteristics using the same hardware and dev setup.
  4. Compare the implementation cost and the resulting runtime behavior.
  5. Derive recommendations for when each pattern is justified.

8. Draft Contribution Statement

This paper contributes a practical evaluation framework for multi-framework micro front end architectures and demonstrates that interoperability strategy, not just module federation itself, is a primary determinant of runtime simplicity, reusability, and developer experience.
